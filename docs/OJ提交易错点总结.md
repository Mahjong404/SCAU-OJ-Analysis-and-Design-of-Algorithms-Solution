# OJ 提交易错点总结

> 记录提交 OJ 过程中发现并修复的所有错误，避免重复踩坑。
> 最后更新：2026-06-04

## 第二单元 递归和分治

### 8594 有重复元素的排列问题

**错误现象**：输出顺序与 OJ 期望不一致（第一个测试用例就失败）

**根因**：代码对输入字符做了 `sort(s, s+n)`，导致输出按字典序排列。但 OJ 要求按课本例 2-4 的算法输出顺序（即原始输入顺序），不排序。

**修复**：删除 `sort(s, s+n)` 调用。

**易错点**：
- 排列/组合生成类题目，**输出顺序必须与课本算法严格一致**，不能自行排序
- 课本例 2-4 的算法对重复元素使用 `findSame(k, i)` 跳过，其输出顺序取决于输入顺序

## 第三单元 动态规划

### 10303 数字三角（优先做）

**错误现象**：正确率 50%，某个测试用例通过但另一个失败

**根因**：
1. 代码有两段路径回溯逻辑，第一段（条件 `<`）是死代码，第二段（条件 `<=`）覆盖了结果
2. 最后输出的语句 `cout << path[i] << (i < n ? ' ' : '\0')` 输出 `\0`（空字符）而非换行，对所有用例（含通过用例）都不规范

**修复**：精简为单循环回溯，直接在循环内输出，去掉冗余的 `path` 数组

**易错点**：
- 回溯路径时注意**"靠右优先"**的含义：dp 值相等时选择右下方向
- OJ 对输出字符逐字节比较，`\0` 虽不可见但会影响结果

### 11090 最大 m 段乘积和最小 m 段和（优先做）

**错误现象**：正确率 66%，第 7 个用例失败

**根因**：输出表达式时使用 `S.substr(k, i-k)` 直接输出子串，保留了前导零。例如输入 `2000000009` 分成 `2` 和 `000000009`，输出了 `2+000000009=11` 而非 `2+9=11`。

**修复**：打印每段时用 `stoi(S.substr(...))` 转为整数再输出，自动消除前导零。

**易错点**：
- 题目明确规定"**若分出的某段数字有 0 开头的，不输出 0**"——必须用整数值输出
- `print` 函数递归输出时，基 case（j==1）也要转 `stoi`，不能遗漏

### 17089 最大 m 子段和

**错误现象**：正确率 0%，第 2 个用例输出 24，期望 22

**根因**：
1. `c[j-1] = 0`（对 j>1 也适用）允许了**空子段**（和为 0）的存在，导致算法可以只用 j-1 个非空子段（补一个空的）就达到更高和
2. `ans = max(ans, maxEnd)` 取了所有 j 值的最大值，但题目要求**恰好 m 个子段**

**修复**：
- `c[j-1] = (j==1) ? 0 : -1e9`，禁止 j>1 时空段
- `ans = maxEnd` 只用最后一轮（j=m）的结果

**易错点**：
- 课本 P59 优化代码只初始化了 `c[1]=0`，未初始化的 c 元素会导致错误
- "空子段"的初始值 0 只有在 j=1（Kadane 启动）时有意义
- "当 m 个子段的每个子段和都是负值时，定义 m 子段和为 0"——这仅用于最终答案，不能用于中间过程

## 第四单元 贪心算法

### 8598 整除 15 问题

**错误现象**：超时（无限循环输出）

**根因**：调整 sum%3 时删除了唯一的末尾数字（0 或 5），导致后续 `while(cnt[i]--)` 中 cnt[5] 变为 -1，进入死循环。

具体场景：输入 `54676778`，sum%3=2，需删除一个余 2 的数。唯一的 5 被删除后，没有 0 或 5 能放在末尾。

**修复**：
1. 调整 sum%3 时，若 5 是唯一末尾数，优先删除其他余 2 的数（2、8）
2. 调整后加事后验证：重新计算 sum%3==0，不通过则输出 `impossible`

**易错点**：
- 贪心删除数字时，**必须保护满足整除 5 条件的关键数字**（0 或 5）
- 删除逻辑要区分 `only5`（cnt[0]==0 && cnt[5]==1）场景
- 删除后**必须验证 sum%3==0**，因为受保护数字的限制可能导致删不够
- `cnt[i]--` 后可能变负 → `while(cnt[i]--)` 死循环

### 10346 带价值的作业安排问题

**错误现象**：答案错误

**根因**：原算法把 deadline 理解为"在 deadline 之前任意时刻完成"（向后搜索空闲槽位），但 OJ 把 deadline 理解为**固定的时间点**（一个时间点只能做一个作业）。

**修复**：改为 `if (!used[j[i].d]) { used[j[i].d] = true; total += j[i].p; }`，仅在 deadline 时刻有空才调度。

**易错点**：
- 仔细读题：**deadline 是固定时刻还是截止时间**，算法完全不同
- 固定时刻 → 按利润排序后直接检查时刻是否被占用
- 截止时间 → 按利润排序后从 deadline 向前搜索空闲槽

## 第五单元 搜索算法

### 17081 宝石排列

**错误现象**：答案错误

**根因**：算法只区分了宝石的形状和颜色两类属性，但没有跟踪**每个具体的宝石**是否已被使用。多颗相同形状颜色的宝石应被视为不同的个体，不能重复使用。

**修复**：添加 `gemUsed[s][cl]` 二维数组，标记每颗具体宝石的使用状态。

**易错点**：
- 回溯搜索中，**同类物品不等于同一物品**，需按个体标记 used
- 尤其注意输入中同一形状+颜色组合可能有多颗

## 自动化脚本

### auto_submit.mjs — 正则匹配失败

**错误现象**：精确匹配（用户+题目）总是失败，总是 fallback 到关键词检测

**根因**：正则 `(\d+)\s+USER\s+\S+\s+PROBLEM\s+(\S+)` 要求用户和题目之间的昵称字段至少一个非空字符（`\S+`），但账号没有设置昵称（空字段），导致 `\S+` 消耗了题目编号。

**修复**：改为 `(\d+)\s+USER\s+(?:\S+\s+)?PROBLEM\s+(\S+)`，昵称字段变为可选组。

**易错点**：
- OJ 的"最近提交列表"是全局共享的，**必须用用户名+题号双重匹配**
- 用户昵称可能为空，正则要用 `?` 而非 `+` 处理可选列
